# 《World of Zuul: Uncertain Fate》游戏需求分析文档

## 1. 开发目标 (Development Objectives)

本项目基于经典 Java 文本冒险游戏 `World of Zuul` 进行深度扩展与现代化改造，旨在交付一个兼具叙事深度、视觉表现与规范工程创新的网页端游戏作品。

- **核心交付转型**：项目最终目标从传统的“Java 命令行文本冒险游戏”彻底调整为“基于 Spring Boot 与 React + JS 的前后端分离浏览器网页端叙事解谜 RPG”。最终展示、答辩和玩家体验均以浏览器 Web 页面为主。
- **交互体验升级**：玩家无需再在命令行中背诵和输入命令。探索、解谜、背包管理、查看世界地图、参与战斗及小游戏等核心交互，全部通过网页端直观的 **UI 按钮、画面点击、弹窗交互和输入面板**完成。
- **工程与协同规范**：基于 2 人协同开发背景，利用 GitHub Issues 驱动开发，规范控制代码冲突边界。引入 GitHub Actions 实现持续集成（CI），通过自动化测试保障代码质量，确保前后端能够高效、高质量地集成交付。

## 2. 游戏设定 (Game Settings)

- **故事背景**：Zuul 世界原本由包含“记忆 (Memory)”、“秩序 (Order)”与“灵魂 (Soul)”的**命运核心 (Fate Core)** 所稳定。然而，邪恶的“王座主宰 (Zuul Overlord)”利用命运骰子污染了核心，导致整个世界陷入了无尽的死亡轮回（Loop）之中。

- **玩家成长路径**：玩家将扮演一名打破死局的挑战者，沿袭以下路径觉醒：

  $$\text{挑战者 (Challenger)} \longrightarrow \text{解谜者 (Puzzle Solver)} \longrightarrow \text{反抗者 (Rebel)} \longrightarrow \text{循环打破者 (Cycle Breaker)} \longrightarrow \text{章节谱写者 (Chapter Writer)}$$

- **核心主题**：游戏不鼓励世俗的称王或纯粹的毁灭。真正的终局是：**“你不登上王座，也不毁灭世界；你终结轮回，并开始谱写属于你自己的篇章。”**

- **反循环关键道具 (Anti-Cycle Key Items)**：

  玩家在世界各地达成特定条件（如解谜或小游戏大成功）可以获得抗御轮回的 5 件核心道具：

  1. `blank_dice` (空白骰子) —— 破损书架真相线索
  2. `savebreaker_key` (碎档钥匙) —— 镜像大成功或碎镜室
  3. `nameless_badge` (无名徽章) —— 三重封印门
  4. `pure_seed` (纯净种子) —— 灵魂花园大成功
  5. `throne_fragment` (王座碎片) —— 击败最终 Boss Zuul Overlord

- **最终选择与多结局**：击败最终 Boss 后，系统根据玩家当前背包中**反循环关键道具的数量**动态解锁对应选项（共 6 个阶梯式结局）。若集齐 5 件道具并选择 `write_own_chapter`（谱写新章），则达成真结局，正式解锁 **Creator Mode (创作者模式)**。

## 3. 开发框架 (Development Framework)

项目采用现代**前后端分离（Decoupled）架构**搭建，提供高效的开发协同体验：

### 3.1 后端架构 (Backend)

- **核心框架**：**Spring Boot**（利用其强大的依赖注入、开箱即用的 Web 组件和自动配置能力，快速构建稳健的 RESTful 服务）。
- **数据处理**：Spring Boot 内置的 Jackson（用于解析和自动序列化自定义章节 JSON 及玩家存档持久化）。
- **项目管理**：Maven 3.8+（集成自动化集成测试与打包插件）。

### 3.2 前端架构 (Frontend)

- **核心框架**：**React (JavaScript)**
- **构建工具**：Vite（提供快速的热更新与高效的前端打包能力）。
- **样式管理**：CSS Tokens / 模块化 CSS（统一暗色奇幻美学，如低饱和深蓝灰、暗紫、黑金、青白光等）。

### 3.3 通信协议

前端不进行任何游戏规则或数据状态的计算。前后端通过统一的 **HTTP JSON API** 进行通信：

- 后端统一向前端暴露游戏状态快照（`GameSnapshot`）。
- 前端组件基于快照中的可用动作列表（`availableActions`）动态渲染交互按钮。玩家点击后发送请求，后端计算并返回更新后的最新快照。

## 4. 开发模块及实施顺序 (Development Modules & Process)

为了有效控制 2 人小组在协同开发时的代码冲突边界，文档内容严格按照**实际开发时间顺序**划分为以下四个阶段进行演进：

### 阶段 1：确定项目框架与引入依赖 (Framework & Dependencies)

本阶段的核心任务是搭建项目的骨架，配置基础环境，确保前后端工程能够各自独立独立启动并引入必要依赖。

- **后端 (Spring Boot 框架搭建)**：
  - 初始化 Spring Boot 项目，引入 `spring-boot-starter-web` 依赖以支持 RESTful API 开发。
  - 引入 `spring-boot-starter-test` 确保单元测试环境就绪。
  - 配置 Maven 编译插件及打包插件，配置基本的项目目录结构（如 `controller`、`service`、`model` 等）。
- **前端 (React/Vite 框架搭建)**：
  - 使用 Vite 脚手架初始化 React (JavaScript) 项目，理清 `src/components` 和 `src/assets` 目录。
  - 引入 `axios` 用于后续的 HTTP 请求通信，引入项目所需的暗色系基础 CSS 样式变量。

### 阶段 2：完成接口设计与数据契约 (API Design & Data Contract)

在本阶段，前后端开发者共同制定并锁定 API 交互协议，以防后续并行开发时出现数据格式不对齐的问题。

- **定义核心游戏快照结构**：
  - 设计统一的响应对象 `GameSnapshot`，包括：当前房间 ID、房间描述、玩家血量（HP）、背包列表（Inventory）、当前游戏阶段（`GamePhase`，如探索、战斗、小游戏、结局选择等）、以及当前允许玩家进行的动作列表（`availableActions`）。
- **锁定核心 API 路由**：
  - `GET /api/game/init`：初始化/新游戏，返回初始快照。
  - `POST /api/game/action`：接收玩家交互请求（如移动、查看、解谜、战斗动作），返回计算后的最新快照。
  - `POST /api/game/save` / `POST /api/game/load`：游戏存档与读档接口。
- **Mock 数据配置**：前端根据约定的 JSON 结构建立 Mock 数据，后端编写基础 Controller 框架，确保接口骨架畅通。

### 阶段 3：功能模块的具体实现 (Core Modules Implementation)

此阶段为核心业务逻辑编写阶段，前后端根据职责划分并行开发具体的功能模块。

- **后端模块具体实现 (Spring Boot Services)**：
  - **空间与地图模块 (`RoomService`)**：实现 10 余个奇幻房间的拓扑连接与动态描述文本切换逻辑。
  - **物品与合成模块 (`ItemService` / `PlayerService`)**：实现背包防止重复获取关键道具的机制，以及通过材料合成 `soul_bell`（灵魂之铃）的系统。
  - **谜题与事件引擎 (`PuzzleEngine` / `EventEngine`)**：编写密码、方向序列等四大类谜题的判定逻辑，实现解谜成功后触发 Flag 变更和道具发放。
  - **互动小游戏模块 (`MiniGameService`)**：实现骰子、点数、连线三个小游戏的独立逻辑与状态 Session 管理。
  - **战斗与结局分支模块 (`BattleService` / `EndingService`)**：编写 Boss 战数值公式、Phase 2 状态转换，以及根据反循环道具数量动态提供结局选项的逻辑。
  - **扩展创客模块 (`CustomChapterService`)**：实现外部 `custom-chapters/` 目录下的 JSON 自定义章节读取、强类型错误码校验，以及旧日遗留 Boss（`legacy-boss.json`）的封存机制。
- **前端组件具体实现 (React Components)**：
  - 依照高内聚原则，独立开发基础组件：
  - `MainMenu`（主菜单）、`GameScreen`（探索大厅与场景叙事）、`DirectionButtons`（方向移动控制）。
  - `InventoryModal`（背包弹窗）、`WorldMapModal`（已探索区域地图展示）。
  - `PuzzleModal`（谜题交互输入）与 `MiniGamePanel`（小游戏可视化面板，包含掷骰结果反馈和卡牌点数锁定等）。
  - `BattlePanel`（包含 Boss 立绘、动态血条、回合特性展示）与 `ChoicePanel`（多结局命运选择面板）。
  - `CreatorModePanel`（创作者模式，支持动态加载和配置自定义章节 UI）。

### 阶段 4：前端对齐与联调测试 (Integration & Debugging)

本阶段将前端 React 界面与 Spring Boot 后端真实业务逻辑进行对接，全面开展集成测试与细节打磨。

- **数据联调与对齐**：
  - 移除前端的 Mock 数据，配置 Vite 反向代理（Proxy），将前端的 Axios 请求全面导向 Spring Boot 后端端口。
  - 验证前端组件是否能根据后端 `GameSnapshot` 返回的 `availableActions` 动态、正确地渲染出对应的 UI 按钮与弹窗。
- **业务流与异常测试**：
  - 联调完整的游戏主线流程：主菜单 $\rightarrow$ 场景探索 $\rightarrow$ 解谜/小游戏 $\rightarrow$ 材料合成削弱 Boss $\rightarrow$ 战胜 Overlord $\rightarrow$ 根据道具数解锁多结局选择。
  - 测试异常情况：如网络延迟、非法操作请求（发送了当前状态下不可用的 action 时的后端拦截与前端友好提示）。
- **自动化构建与总验收**：
  - 配置前后端工程的统一自动化构建流。
  - 运行 Maven 整体集成测试，确保后端测试覆盖率达标，前端能够顺利完成生产环境打包（`npm run build`），项目整体达到答辩与最终交付标准。
